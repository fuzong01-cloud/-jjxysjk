# 系统设计、使用与部署指南

## 数据设计

`students` 以身份证号作为唯一业务标识，与 `education_records` 一对一，与荣誉、经营主体和培育记录一对多；经营主体再与年度营收和主营产业一对多。导入批次与错误、管理员、审计日志、备份记录独立保存。金额使用 `Numeric(18,2)`，日期使用 `Date`，年龄由出生日期实时计算。

常用索引覆盖身份证号、姓名、区县、手机号、经营主体名称、统一社会信用代码及审计时间。业务记录保留软删除字段，关联明细使用外键级联。

## Excel 导入

仅支持 `.xlsx`。数据工作表优先选择 `Sheet1`，否则选择数据量最大的表。必填列为“学员姓名”和“身份证号”。系统解析基本信息、教育经历、荣誉序号组、经营主体、营收年份组、主营产业组和培育信息。

浏览器导入分两步：上传后生成预览批次，确认统计结果后才写入。重复身份证号更新已有档案；错误行不写入并保留 Excel 行号。未知列记录为警告，不会静默丢弃。

## 管理员操作

1. 用 `.env` 中账号登录。
2. 在“数据导入”上传原始名单，检查新增、更新和错误数量，再确认写入。
3. 在“学员档案”按姓名、区县或经营主体检索，打开档案查看详情或添加荣誉。
4. 使用“导出当前结果”下载 Excel。
5. 在“操作日志”审阅写操作；身份证号和手机号在日志中脱敏。
6. 在“备份恢复”创建或下载数据库备份。恢复会先生成保护性备份并校验数据库结构。

## API

所有业务接口位于 `/api/v1`，采用 Cookie 会话认证。主要分组为 `/students`、`/business-entities`、`/import`、`/export`、`/audit-logs` 和 `/backups`。成功响应使用 `data/message`，列表使用 `items/total/page/page_size`，校验错误使用 `code/message/details`。交互式完整定义见 `/docs`。

## 部署与升级

正式使用前设置随机 `APP_SECRET_KEY` 和强管理员密码，并限制数据库、备份目录的 Windows 文件权限。默认命令只监听 `127.0.0.1`；若开放局域网，应置于 HTTPS 反向代理后，并将 Session Cookie 的 `https_only` 改为 `True`。

升级前先备份数据库，再执行 `python -m alembic upgrade head`。切换 PostgreSQL 时安装对应驱动、修改 `DATABASE_URL`，在测试环境执行迁移和导入验收后再切换。

## 开发约定与排错

Python 使用类型标注，数据库字段使用 snake_case，业务规则集中在服务层，所有修改操作写审计。新增字段时同步修改模型、迁移、导入映射、接口模式及测试。

- 无法登录：检查 `.env` 是否在首次启动前修改；已建库后修改密码不会自动覆盖管理员。
- Excel 导入失败：确认文件不是 Excel 打开的临时文件（文件名以 `~$` 开头），并查看批次错误报告。
- 数据库被占用：停止运行中的服务后再执行离线恢复。
- 忘记密码：先备份数据库，再通过受控维护脚本重置，不要直接在日志或文档中保存明文密码。
